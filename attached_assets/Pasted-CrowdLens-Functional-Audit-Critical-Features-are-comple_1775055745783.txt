CrowdLens Functional Audit
🔴 Critical — Features are completely broken
1. Simulation mode is dead code
simulation.py has a fully built SimulationEngine with moving people, running behaviour, and object spawning — but main.py never imports or starts it. When the app loads in idle mode, the WebSocket broadcasts nothing. The dashboard is completely empty. First-time users see zero tracks, zero stats, and a static canvas. This is the biggest gap.

2. WebSocket sends nothing without an active source
The /ws endpoint just waits for incoming text from clients and never broadcasts on its own. Data only flows when a video/webcam/stream processing loop is running. Without the simulation connected, users who haven't picked a source see a frozen UI with all stats at 0 indefinitely.

3. useSimulation reconnect memory leak
When the WebSocket hook unmounts (e.g., navigating away), the cleanup runs and calls ws.close(). This fires onclose, which schedules a new reconnect setTimeout — but cleanup has already run, so that timer is never cleared. The hook keeps attempting reconnects forever in the background after the component is gone.

🟠 Major — Features partially work or behave incorrectly
4. Settings save has no error handling
handleSave() in Settings always shows "Applied!" regardless of whether the backend actually accepted the request. If the backend is down or returns an error, the user thinks it worked.

5. Mode switching leaves stale frame data on screen
When the user switches from webcam → idle, the last received WebSocket frame stays in state. Old bounding boxes and anomaly overlays keep rendering on the canvas with no indication they're stale. There's no frame flush or disconnection feedback on the main canvas.

6. Webcam bounding box misalignment on non-16:9 cameras
The canvas draws the raw local webcam feed, which preserves whatever aspect ratio the camera provides. The backend resizes every frame to 1280×720 before YOLO runs. If the camera isn't 16:9 (e.g., some laptops output 4:3), the stretch ratio on the canvas is different from what YOLO computed coordinates against — bounding boxes will be offset.

7. Test stream URL is hardcoded to localhost
The "Fill built-in test stream URL" button inserts http://localhost:8080/api/stream/test-feed. This URL is sent to the backend which then tries to open it with cv2.VideoCapture. The backend calling its own MJPEG streaming endpoint is a circular connection that is fragile — the stream loop blocks an executor thread reading a response from the same Uvicorn process.

8. Alert sound tracks anomaly types, not individual tracks
useAlertSound only plays a sound when a new anomaly type appears that wasn't in the previous frame. If running is already active and a second person starts running, no sound plays. Also, when switching modes, the "last seen types" set carries over, so the first alert of a new session might be silently swallowed.

🟡 Minor — Polish and UX issues
9. avg_speed shown without rounding in Alert History
Speeds like 23.4567891 are displayed raw in the details column. Should be rounded to 1–2 decimal places.

10. JSON parse errors silently swallowed in useSimulation
The onmessage handler wraps parsing in try {} catch {} with an empty catch. Malformed frames are dropped with no logging, making WebSocket issues impossible to debug in the browser.

11. No stale/disconnected indicator on the main canvas
The connected state is visible in the sidebar, but the canvas itself has no overlay when the WebSocket is disconnected. Users might think detection is running when it's actually severed.

12. Video file stays "ready" across sessions
After uploading and stopping a video, videoStatus.filename persists until the next upload or server restart. If the user refreshes, it still shows the old file as ready to process, which can be confusing.
